## 上下文

### 问题背景
NOTAM（航行通告）是航空领域的重要信息源，但存在以下痛点：
1. **高度专业化编码** — 大量缩写、代码、术语，非专业人士难以解读
2. **语言障碍** — 国际 NOTAM 以英语发布，非英语母语人员理解困难
3. **信息过载** — 数量庞大，包含大量例行、重复或无关信息
4. **非结构化文本** — 电报格式可读性差，关键信息淹没在文本中

### 用户需求
- **具体用户**：已有明确用户在等待（非假设需求）
- **当前方案**：用户有临时方案在撑着（问题足够痛）
- **交付形式**：Python API 服务，输出结构化 JSON

## Outside Voice (Codex/Claude Subagent)

评审完成，发现以下问题：

### LOGICAL GAPS
1. **Q 行代码解码缺失** - 计划提到"提取 Q 行结构化字段"，但 NOTAM 的 Q 行是编码字符串（如 `Q)EGTT/QFALC/IV/NBO/A/000/999/...`），需要专门的解码逻辑将每个段映射到语义（飞行情报区、NOTAM 代码、交通类型等）。
2. **E 行解析输出一致性** - LLM 层同时输出 5 种结果，未定义当 LLM 对同一内容给出矛盾信息时的处理逻辑。
3. **限制区域解析边界** - 未说明如何处理非标准格式的限制区域描述（自由文本、多边形坐标、高度层混合表达）。
4. **错误恢复策略不完整** - `PARSE_WARNING` 标记后，该 NOTAM 是否继续进入 LLM 层？

### OVERCOMPLEXITY
1. **PostgreSQL 可能过度设计** - 若核心功能是无状态 API，为何需要 PG 持久化？若只需要任务状态追踪，SQLite 或纯 Redis 可能足够。
2. **RQ + Redis 任务队列** - 对于内网服务，若并发量不高，同步处理 + 超时控制可能更简单。
3. **批量 + 异步双模式** - 未定义 API 何时走同步路径、何时走异步路径。

### FEASIBILITY RISKS
1. **LLM JSON 输出可靠性** - 3 次重试后仍失败，API 返回什么？
2. **术语解释的边界** - 使用 LLM 内生知识还是外挂术语库？纯 LLM 可能产生幻觉解释。
3. **翻译质量责任** - 航空翻译错误可能导致严重后果，需校验机制或免责声明。
4. **速率限制与成本** - 未估算 NOTAM 处理量与 LLM API 成本。

### MISSING DEPENDENCIES/SEQUENCING
1. **NOTAM 样本依赖** - 实现正则解析器需要真实 NOTAM 样本验证格式覆盖。
2. **JSON Schema 下游确认** - 未确认下游系统是否已定义 schema 需求。
3. **内网服务器网络配置** - 若使用 OpenAI API，内网服务器需要出网权限。
4. **Redis/PostgreSQL 运维依赖** - 谁负责部署和维护这些基础设施？

### STRATEGIC MISCALIBRATION
1. **LLM 层是否必要** - 术语解释和翻译可基于规则/词典实现，成本更低且确定性更高。
2. **"完整解析器"定位模糊** - 是实验性工具还是生产系统？计划介于两者之间。
3. **范围 creep 风险** - "限制区域结构化解析"本身可能是一个子项目，建议拆分为 Phase 1（基础解析）和 Phase 2（高级语义）。

## 架构决策（Eng 评审补充）

### 完整架构（Phase 2）

| 决策点 | 选择 | 理由 |
|--------|------|------|
| API 设计 | 批量 + 异步任务队列 | 适合大量 NOTAM 处理 |
| 任务队列 | RQ (Redis Queue) | 轻量级，只需要 Redis |
| 错误处理 | 重试 + 失败标记 | 不丢失数据，可追溯 |
| 存储方案 | PostgreSQL + Redis | PG 持久化历史，Redis 队列 |
| LLM 调用 | 实时调用 | 结果最新，无需缓存管理 |
| 测试框架 | pytest + tests/ 分离 | Python 生态标准 |

### MVP 架构（Phase 1 — 当前实现）

| 决策点 | 选择 | 理由 |
|--------|------|------|
| API 设计 | 同步 API | MVP 最简单路径，直接返回结果 |
| 存储 | SQLite | 零运维，可无缝迁移到 PostgreSQL |
| 任务队列 | 无 | MVP 不需要批量/异步 |
| Q 行处理 | 完整解码 | 飞行情报区、NOTAM 代码、交通类型等 |
| 术语解释 | LLM + 术语库校验 | 平衡准确性和成本 |
| LLM 一致性 | Schema 验证 + 重试 | 确保输出格式稳定 |

## 错误与救援映射

| 代码路径 | 错误类型 | 异常类 | 重试策略 | 用户可见 | 日志 |
|----------|----------|--------|----------|----------|------|
| 正则解析 | 格式不合法 | `NotamParseError` | 无 | 标记为 `PARSE_WARNING` | 完整上下文 |
| 正则解析 | 字段缺失 | `FieldMissingWarning` | 无 | 字段为 null | 警告级别 |
| LLM 调用 | API Timeout | `APITimeoutError` | 指数退避 5 次 | 重试中状态 | 完整堆栈 |
| LLM 调用 | 429 Rate Limit | `RateLimitError` | 指数退避 5 次 | 重试中状态 | 警告级别 |
| LLM 调用 | JSON 解析失败 | `JSONParseError` | 重试 LLM 3 次 | 重试中状态 | 原始响应 |
| DB 操作 | 连接失败 | `DBConnectionError` | 固定间隔 3 次 | `DB_ERROR` | 完整堆栈 |
| DB 操作 | 写入冲突 | `IntegrityError` | 无 | `DB_ERROR` | 完整上下文 |

**CRITICAL GAP**: 所有错误必须记录完整的 NOTAM 原始文本和任务 ID，便于后续调试和重放。
